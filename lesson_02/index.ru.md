**Урок ассемблера FFmpeg № 2**

Теперь, когда вы написали свою первую функцию на ассемблере, мы познакомимся с ветвлениями и циклами.

Сначала нужно ввести понятия меток и переходов. В искусственном примере ниже инструкция jmp переносит выполнение кода на место после «.loop:». «.loop:» называется *меткой*, а точка перед именем метки означает, что это *локальная метка* — это позволяет использовать одно и то же имя метки в разных функциях. Этот пример, конечно, показывает бесконечный цикл, но позже мы превратим его в нечто более реалистичное.

```assembly
mov  r0q, 3
.loop:
    dec  r0q
    jmp .loop
```

Прежде чем написать настоящий цикл, нужно познакомиться с регистром *FLAGS*. Мы не будем углубляться в тонкости *FLAGS* (опять же потому, что операции с GPR — по большей части строительные леса), но есть несколько флагов, таких как флаг нуля (Zero-Flag), флаг знака (Sign-Flag) и флаг переполнения (Overflow-Flag), которые устанавливаются по результату большинства не-mov инструкций над скалярными данными — например, арифметических операций и сдвигов.

Вот пример, где счётчик цикла уменьшается до нуля, а jg (jump if greater than zero — прыгнуть, если больше нуля) служит условием цикла. dec r0q устанавливает флаги по значению r0q после выполнения инструкции, и по ним можно совершать переход.

```assembly
mov  r0q, 3
.loop:
    ; какая-то работа
    dec  r0q
    jg  .loop ; прыжок, если больше нуля
```

Это эквивалентно следующему коду на C:

```c
int i = 3;
do
{
   // какая-то работа
   i--;
} while(i > 0)
```

Такой C-код выглядит немного неестественно. Обычно цикл на C пишут так:

```c
int i;
for(i = 0; i < 3; i++) {
    // какая-то работа
}
```

Это примерно эквивалентно следующему (простого способа точно повторить такой ```for``` нет):

```assembly
xor r0q, r0q
.loop:
    ; какая-то работа
    inc r0q
    cmp r0q, 3
    jl  .loop ; прыжок, если (r0q - 3) < 0, т. е. (r0q < 3)
```

В этом фрагменте стоит отметить несколько вещей. Во-первых, ```xor r0q, r0q``` — распространённый способ обнулить регистр, который на некоторых системах быстрее, чем ```mov r0q, 0```, потому что, попросту говоря, никакой загрузки не происходит. То же самое можно делать и с SIMD-регистрами: ```pxor m0, m0``` обнуляет весь регистр. Во-вторых, обратите внимание на cmp. cmp фактически вычитает второй регистр из первого (никуда не сохраняя результат) и устанавливает *FLAGS*, но, как сказано в комментарии, его нужно читать вместе с переходом (jl = jump if less than zero — прыгнуть, если меньше нуля): прыжок произойдёт, если ```r0q < 3```.

Заметьте, что в этом фрагменте на одну инструкцию больше (cmp). В общем случае меньше инструкций = быстрее код, поэтому предыдущий вариант предпочтительнее. В следующих уроках вы увидите ещё приёмы, позволяющие избежать этой лишней инструкции и заставить арифметику или другую операцию сама установить *FLAGS*. Обратите внимание: мы не пишем ассемблер, точно повторяющий циклы C, — мы пишем циклы так, чтобы они были максимально быстрыми именно в ассемблере.

Вот часто используемые мнемоники переходов (*FLAGS* приведены для полноты, но для написания циклов знать их тонкости не обязательно):

| Мнемоника | Описание | FLAGS |
| :---- | :---- | :---- |
| JE/JZ | Прыжок, если равно / ноль | ZF = 1 |
| JNE/JNZ | Прыжок, если не равно / не ноль | ZF = 0 |
| JG/JNLE | Прыжок, если больше / не меньше-или-равно (знаковое) | ZF = 0 и SF = OF |
| JGE/JNL | Прыжок, если больше или равно / не меньше (знаковое) | SF = OF |
| JL/JNGE | Прыжок, если меньше / не больше-или-равно (знаковое) | SF ≠ OF |
| JLE/JNG | Прыжок, если меньше или равно / не больше (знаковое) | ZF = 1 или SF ≠ OF |

**Константы**

Посмотрим на примеры использования констант:

```assembly
SECTION_RODATA

constants_1: db 1,2,3,4
constants_2: times 2 dw 4,3,2,1
```

* SECTION_RODATA объявляет секцию данных только для чтения. (Это макрос, потому что разные форматы выходных файлов, используемые операционными системами, объявляют её по-разному)
* constants_1: метка constants_1 определена как ```db``` (declare byte — объявить байты), т. е. эквивалент uint8_t constants_1[4] = {1, 2, 3, 4};
* constants_2: здесь используется макрос ```times 2```, повторяющий объявленные слова, т. е. эквивалент uint16_t constants_2[8] = {4, 3, 2, 1, 4, 3, 2, 1};

Эти метки, которые ассемблер превращает в адреса памяти, затем можно использовать в загрузках (но не в сохранениях — данные только для чтения). Некоторые инструкции принимают адрес памяти как операнд, так что их можно использовать без явной загрузки в регистр (у этого есть свои плюсы и минусы).

**Смещения**

Смещение (offset) — это расстояние (в байтах) между соседними элементами в памяти. Смещение определяется **размером каждого элемента** структуры данных.

Теперь, когда мы умеем писать циклы, пора доставать данные. Но здесь есть отличия от C. Посмотрим на такой цикл на C:

```c
uint32_t data[3];
int i;
for(i = 0; i < 3; i++) {
    data[i];
}
```

4-байтовое смещение между элементами data компилятор C вычисляет сам. А при написании ассемблера вручную эти смещения нужно вычислять самому.

Посмотрим на синтаксис вычисления адресов памяти. Он применим ко всем видам адресов:

```assembly
[base + scale*index + disp]
```

* base — GPR (обычно указатель из аргумента C-функции)
* scale — может быть 1, 2, 4, 8. По умолчанию 1
* index — GPR (обычно счётчик цикла)
* disp — целое число (до 32 бит). Displacement — смещение внутри данных

x86asm предоставляет константу mmsize, которая сообщает размер SIMD-регистра, с которым вы работаете.

Вот простой (и бессмысленный) пример, иллюстрирующий загрузку с произвольными смещениями:

```assembly
;static void simple_loop(const uint8_t *src)
INIT_XMM sse2
cglobal simple_loop, 1, 2, 2, src
     movq r1q, 3
.loop:
     movu m0, [srcq]
     movu m1, [srcq+2*r1q+3+mmsize]

     ; какая-то работа

     add srcq, mmsize
dec r1q
jg .loop

RET
```

Обратите внимание: в ```movu m1, [srcq+2*r1q+3+mmsize]``` ассемблер сам предвычислит нужную константу смещения. В следующем уроке мы покажем трюк, позволяющий избавиться от add и dec в цикле, заменив их одним add.

**LEA**

Теперь, когда вы понимаете смещения, вы можете использовать lea (Load Effective Address — загрузить эффективный адрес). Она позволяет выполнить умножение и сложение одной инструкцией, что быстрее, чем несколько инструкций. Разумеется, есть ограничения на то, на что можно умножать и что прибавлять, но это не мешает lea быть мощной инструкцией.

```assembly
lea r0q, [base + scale*index + disp]
```

Вопреки названию, LEA можно использовать и для обычной арифметики, а не только для вычисления адресов. Можно делать даже такое:

```assembly
lea r0q, [r1q + 8*r2q + 5]
```

Заметьте, что содержимое r1q и r2q при этом не меняется. Также не затрагивается *FLAGS* (поэтому прыгать по результату lea нельзя). LEA избавляет от всех этих инструкций и временных регистров (код ниже не полностью эквивалентен, потому что add меняет *FLAGS*):

```assembly
movq r0q, r1q
movq r3q, r2q
sal  r3q, 3 ; арифметический сдвиг влево на 3 = * 8
add  r3q, 5
add  r0q, r3q
```

Вы часто увидите lea при подготовке адресов перед циклами или в вычислениях вроде показанного выше. Конечно, доступны не любые умножения и сложения, но умножение на 1, 2, 4, 8 и прибавление фиксированного смещения — типичный случай.

В задании вам нужно будет загрузить константу и в цикле прибавить её значения к SIMD-вектору.
